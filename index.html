ETAPA 1 DE 6

Importações, Configuração do Backend e Banco de Dados

Antes de criar qualquer rota, o backend precisa importar as bibliotecas certas, configurar a aplicação da web em Flask, e ter um banco de dados já criado e conectado. Essa é a base sobre a qual tudo o resto vai ser construído.

Passo a Passo

Primeiro crie a pasta do projeto e instale as bibliotecas necessárias. Depois crie o arquivo de backend com todas as importações, e configurações do Flask, do CORS e do JWT, sem nenhuma rota ainda. Por fim, crie o banco de dados e o arquivo responsável pela conexão com ele. Isso garante que, ao rodar o servidor da web, ele já inicia sem erros antes de adicionarmos qualquer funcionalidade.

Passo 1 - Crie a pasta do projeto

Abra o VS Code (ou outro editor de código de sua preferência) e crie uma nova pasta vazia chamada de Kanban com SQL Puro. Essa pasta será a raiz do projeto, e é dentro dela que todos os arquivos e subpastas das próximas etapas vão ser criados.

Abrindo a pasta no VS Code No VS Code, vá em Arquivo → Abrir Pasta e selecione a pasta chamada de Kanban com SQL Puro que você acabou de criar. Isso garante que o terminal integrado do editor já abra na pasta correta do projeto.

Passo 2 - Crie o arquivo de backend (app.py)

Na raiz do seu projeto, crie um arquivo chamado de app.py com o conteúdo abaixo:

python — app.py

# Biblioteca para criptografar senhas.
from werkzeug.security import generate_password_hash, check_password_hash

# Biblioteca para interação do frontend com o backend.
from flask_cors import CORS

# Biblioteca para autenticação do Token da Web (JWT).
from flask_jwt_extended import JWTManager, create_access_token, jwt_required, get_jwt_identity, get_jwt

# Biblioteca para manipular datas.
from datetime import timedelta

# Módulo de conexão com o banco de dados.
from database import database_connection

# Biblioteca para leitura de variáveis de ambiente.
import os
from dotenv import load_dotenv

# Captura as variáveis de ambiente.
load_dotenv()

# Inicia a aplicação da web.
app = Flask(__name__)

# Permite que sejam feitas requisições de fora.
CORS(app)

# Configurações do Token da Web (JWT).
app.config["JWT_SECRET_KEY"] = os.getenv("JWT_SECRET_KEY")
app.config["JWT_ACCESS_TOKEN_EXPIRES"] = timedelta(hours=8)

# Inicializa o gerenciador de Tokens da Web (JWT).
jwt = JWTManager(app)


# Executa a aplicação da web com modo de depuração ativo.
if __name__ == "__main__":
    app.run(debug=True)

Passo 3 - Instale as bibliotecas necessárias

Olhando para as importações do app.py, algumas vêm prontas com o Python (como os e datetime), mas outras precisam ser instaladas manualmente. Com o terminal de comandos do VS Code aberto na pasta do projeto, rode o comando abaixo:

bash
pip install flask flask-cors flask-jwt-extended python-dotenv mysql-connector-python
Por que esses pacotes e não os nomes das importações? O nome que você usa no pip install nem sempre é igual ao nome usado no import. Por exemplo, você instala flask-jwt-extended, mas importa com flask_jwt_extended (com underline). Já mysql-connector-python é instalado com esse nome, mas importado como mysql.connector, que vamos usar no database.py mais adiante. O werkzeug.security não precisa de instalação separada, pois já vem junto com o Flask.

Passo 5 - Crie o arquivo de conexão com o banco de dados (database.py)

Na raiz do seu projeto, na mesma pasta do app.py, crie um arquivo chamado de database.py com o conteúdo abaixo:

python — database.py
# Biblioteca de conexão com o banco de dados.
import mysql.connector

# Biblioteca para leitura de variáveis de ambiente.
import os
from dotenv import load_dotenv

# Carrega as variáveis do arquivo .env.
load_dotenv()

# Função responsável pela conexão com o banco de dados.
def database_connection():

    return mysql.connector.connect(
        host=os.getenv("DB_HOST"),
        user=os.getenv("DB_USER"),
        password=os.getenv("DB_PASSWORD"),
        database=os.getenv("DB_NAME")

Por que um arquivo separado para a conexão com o banco de dados? Centralizar a conexão com o banco de dados em um único arquivo evita repetição de código. Se as credenciais do banco de dados mudarem no futuro, a alteração é feita em um único lugar, em vez de em cada rota do app.py.

ETAPA 2 DE 6

Login e Logout

Com o backend configurado, vamos criar a primeira funcionalidade real do To do List - Kanban, que é a autenticação de usuário. O usuário envia CPF e senha, o backend verifica no banco de dados e devolve um Token da Web (JWT) que autoriza o acesso às próximas rotas.

Passo a Passo

Primeiro vamos criar as rotas de login e logout de usuário no backend, depois as telas de login e do painel de gerenciamento no frontend, e por fim o JavaScript que conecta tudo isso.

Passo 1 - Crie a rota de login de usuário no backend

No arquivo chamado de app.py, logo após a linha jwt = JWTManager(app), adicione a rota exibida abaixo:

python — app.py
# -------------------------
# Rota para fazer login.
@app.route("/api/login", methods=["POST"])
def login():

    # Recebe os dados do frontend.
    dados = request.get_json()
    cpf = dados.get("cpf")
    senha = dados.get("senha")

    # Abre uma conexão com o banco de dados.
    link = database_connection()
    cursor = link.cursor(dictionary=True)

    # Roda o SQL para buscar os dados do usuário.
    cursor.execute("SELECT * FROM usuario WHERE cpf = %s", (cpf,))
    usuario = cursor.fetchone()

    # Fecha a conexão com o banco de dados.
    cursor.close()
    link.close()

    # Verifica se o usuário existe e se a senha está correta.
    if not usuario or not check_password_hash(usuario["senha"], senha):

        return jsonify({"erro": "CPF ou senha inválidos..."}), 401

    # Gera o token da web (JWT) com o ID do usuário.
    token = create_access_token(
        identity=str(usuario["id"]),
        additional_claims={
            "administrador": bool(usuario["administrador"])
        })

    return jsonify({"token": token}), 200

Passo 2 - Crie a rota de logout de usuário no backend

Logo abaixo da rota de login de usuário, adicione o conteúdo exibido abaixo:

python — app.py
# -------------------------
# Rota para fazer logout.
@app.route("/api/logout", methods=["POST"])
@jwt_required()
def logout():

    return jsonify({"mensagem": "Logout realizado com sucesso!"}), 200

Passo 3 - Compreender as duas rotas criadas

  • cursor(dictionary=True) faz o cursor retornar os dados como dicionário (usuario["cpf"]), em vez de tupla (usuario[0]).
  • check_password_hash compara a senha digitada com o hash que está armazenado no banco de dados.
  • identity=str(usuario["id"]) é obrigatório ser uma string simples. O Flask-JWT-Extended não aceita um dicionário inteiro como identidade do token.
  • additional_claims guarda informações extras dentro do token da web, como se o usuário é administrador, e pode ser lido em qualquer rota protegida.
  • @jwt_required() na rota de logout exige que um token da web válido seja enviado na requisição. Sem ele, o backend retorna erro 401 automaticamente.
  • 401 é o código de HTTP usado quando as credenciais estão incorretas (não autorizado).
Por que o logout não apaga nada no banco de dados? Com JWT (Token da Web), a invalidação do token acontece no navegador do usuário, não no servidor. Por isso a rota de logout é tão simples, ela só confirma que recebeu um token da web válido. Quem realmente desconecta o usuário é o frontend, apagando o token da web salvo.

Passo 7 - Crie o JavaScript que envia os dados de login de usuário

Na pasta de js/ que está dentro de static/, crie um arquivo chamado de index.js com o conteúdo completo abaixo:

javascript — index.js
// Função para fazer o login.
async function fazer_login() {

    // Captura os dados do formulário.
    const cpf = document.getElementById("cpf").value;
    const senha = document.getElementById("senha").value;
    const mensagem = document.getElementById("mensagem");

    try {

        // Envia os dados para o backend.
        const resposta = await fetch("http://127.0.0.1:5000/api/login", {
            method: "POST",
            headers: { "Content-Type": "application/json" },
            body: JSON.stringify({ cpf, senha })
        });

        const dados = await resposta.json();

        // Verifica se o login foi bem-sucedido.
        if (resposta.ok) {

            // Salva o token no navegador da web e redireciona para o painel de gerenciamento.
            localStorage.setItem("token", dados.token);
            window.location.href = "dashboard.html";

        } else {

            // Exibe a mensagem de erro retornada pelo backend.
            mensagem.className = "alert alert-danger";
            mensagem.textContent = dados.erro;

            // Remove a mensagem de erro após 4 segundos.
            setTimeout(function () {
                mensagem.className = "";
                mensagem.textContent = "";
            }, 4000);

        }

    } catch {

        // Exibe a mensagem de erro caso o backend não esteja disponível.
        mensagem.className = "alert alert-warning";
        mensagem.textContent = "Não foi possível atender sua requisição no momento. Tente mais tarde...";

        // Remove a mensagem de erro após 4 segundos.
        setTimeout(function () {
            mensagem.className = "";
            mensagem.textContent = "";
        }, 4000);

    }

}
Atenção ao caminho do redirecionamento O window.location.href = "dashboard.html" funciona corretamente porque o index.html e o dashboard.html estão dentro da mesma pasta de templates/. Se você colocar ../templates/dashboard.html aqui, o caminho vai ficar errado, pois o navegador da web vai tentar sair da pasta atual e entrar em templates/ de novo.

Passo 8 - Crie o JavaScript que protege o painel de gerenciamento e que faz o logout

Na pasta de js/ que fica dentro de static/, crie um arquivo chamado de dashboard.js com o conteúdo completo abaixo:

javascript — dashboard.js
// Verifica se o usuário está logado ao carregar o painel de gerenciamento.
if (!localStorage.getItem("token")) {

    // Redireciona para o login.
    window.location.href = "index.html";
}

// Função para fazer o logout.
async function fazer_logout() {

    // Recupera o token salvo no navegador da web.
    const token = localStorage.getItem("token");

    // Envia a requisição de logout para o backend.
    await fetch("http://127.0.0.1:5000/api/logout", {
        method: "POST",
        headers: {
            "Content-Type": "application/json",
            "Authorization": `Bearer ${token}`
        }
    });

    // Remove o token do navegador da web e redireciona para o login.
    localStorage.removeItem("token");
    window.location.href = "index.html";

}

Passo 9 - Compreender a proteção do painel de gerenciamento

  • if (!localStorage.getItem("token")) roda imediatamente quando o arquivo é carregado, antes de qualquer outra coisa na página de HTML. Se não houver token da web salvo, o usuário é redirecionado de volta para o login, impedindo o acesso direto ao painel de gerenciamento.
  • Authorization: Bearer ${token} é o formato padrão exigido pelo Flask-JWT-Extended para receber o token da wen nas requisições protegidas.
  • O logout remove o token da wen do navegador independentemente da resposta do backend, garantindo que o usuário sempre seja desconectado.

Passo 5 - Compreender as quatro rotas criadas

  • get_jwt()["administrador"] é usado em todas as rotas para garantir que apenas usuários administradores possam gerenciar outros usuários.
  • SELECT id FROM usuario WHERE cpf = %s OR email = %s evita que dois usuários sejam cadastrados com o mesmo CPF ou e-mail, retornando 409 (conflito) caso já exista um registro igual no banco de dados.
  • Na rota de edição de usuário, a verificação de duplicidade usa AND id != %s, para não bloquear a edição de um usuário comparando ele com ele mesmo.
  • Na rota de edição de usuário, a senha só é atualizada se uma nova senha foi enviada. Isso evita que o usuário precise digitar a senha de novo só para editar o nome ou o telefone, por exemplo.
  • 403 é o código de HTTP usado quando o usuário está autenticado, mas não tem permissão para aquela ação (acesso negado).

async function salvar_usuario() {

    // Captura os dados do formulário.
    const id = document.getElementById("id-usuario").value;
    const nome = document.getElementById("nome").value;
    const cpf = document.getElementById("cpf").value;
    const email = document.getElementById("email").value;
    const senha = document.getElementById("senha").value;

        return;

                "Authorization": `Bearer ${localStorage.getItem("token")}`
            },

        // Verifica se a operação foi bem-sucedida.

                mensagem.className = "";
                mensagem.textContent = "";
            }, 4000);

        mensagem.className = "alert alert-warning";

  • arquivo.filename.lower().endswith(".pdf") valida a extensão do arquivo, mas não o conteúdo real dele. Isso significa que um arquivo renomeado para .pdf ainda passaria pela validação, mas é uma limitação aceitável para fins didáticos.
  • send_file(BytesIO(...)) permite enviar o arquivo como download diretamente dos bytes guardados no banco de dados, sem precisar salvar nada em disco antes.

// Função para baixar o arquivo de uma tarefa.
async function baixar_arquivo(id) {

    try {

        const resposta = await fetch(`http://127.0.0.1:5000/api/tarefas/${id}/arquivo`, {
            method: "GET",
            headers: {
                "Authorization": `Bearer ${localStorage.getItem("token")}`
            }
        });

        if (!resposta.ok) {
            const dados = await resposta.json();
            alert(dados.erro);
            return;
        }

        // Converte a resposta em um arquivo e dispara o download no navegador.
        const blob = await resposta.blob();
        const url = window.URL.createObjectURL(blob);
        const link = document.createElement("a");
        link.href = url;
        link.download = "arquivo.pdf";
        link.click();
        window.URL.revokeObjectURL(url);

    } catch {
        alert("Não foi possível atender sua requisição no momento. Tente mais tarde...");
    }

}

Por que não usar Content-Type aqui? Diferente de usuarios.js e status_tarefa.js, este fetch não define "Content-Type": "application/json". Quando o corpo da requisição é um FormData, o próprio navegador da web define o cabeçalho correto automaticamente, incluindo um identificador especial necessário para o backend separar corretamente o arquivo dos demais campos.
Por que baixar_arquivo() usa fetch em vez de um link direto? A rota de download exige o cabeçalho Authorization com o token da web, e um link comum de HTML não tem como enviar esse cabeçalho. Por isso o fetch busca o arquivo, transforma a resposta em blob, cria uma URL temporária na memória do navegador, e simula um clique para disparar o download.

// Função para montar o quadro Kanban com base nos status e nas tarefas cadastradas.
async function montar_quadro_kanban() {

    try {

        // Busca os status de tarefas e as tarefas no backend.
        const resposta_status = await fetch("http://127.0.0.1:5000/api/status-tarefa", {
            method: "GET",

            method: "GET",

        const status_tarefas = await resposta_status.json();
        const tarefas = await resposta_tarefas.json();

        // Atualiza o cartão de total de tarefas.
        document.getElementById("total-tarefas").textContent = tarefas.length;

        const quadro = document.getElementById("quadro-kanban");

        status_tarefas.forEach(function (status) {

            const tarefas_do_status = tarefas.filter(function (tarefa) {

            // Monta os cartões de tarefa dessa coluna.
            let cartoes = "";

                        <p class="text-muted small mb-0">${tarefa.descricao || "-"}</p>
                    </div>
                `;

            quadro.innerHTML += `
                <div class="coluna-kanban">
                    <h6>${status.nome} <span class="badge bg-secondary">${tarefas_do_status.length}</span></h6>

Atenção ao nome do arquivo salvo O navegador da web pode sugerir salvar o arquivo com a extensão .png, mesmo que o conteúdo interno seja um JPEG, isso não é um problema, pois o <img> do HTML reconhece a imagem normalmente pelo seu conteúdo, não pela extensão do nome. O importante é manter o nome do arquivo (incluindo espaços e hífen) exatamente como sugerido pelo botão de download, para bater com os caminhos já usados nas tags de <img> desde a etapa 2.
